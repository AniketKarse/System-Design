# 🆔 Distributed ID Generation

The fundamental challenge is creating a function that returns a **globally unique ID** every time it is invoked across a massive cluster of machines.

---

## 🏗️ Phase 1: Application-Level IDs

If we generate IDs locally within our service, we must avoid **collisions** (two machines/threads creating the same ID at the same microsecond).

### The Evolution of the ID Structure:
1.  **Time Only:** `get_epoch_ms()`
    * *Issue:* Works for 1 machine, but 2+ machines will inevitably collide.
2.  **Machine ID + Time:** `concat(machine_id, time)`
    * *Issue:* Solves machine collisions, but fails if two threads on the same machine call it simultaneously.
3.  **The Counter Approach:** `concat(machine_id, time, counter++)`
    * *The Problem:* The counter is in memory. If the machine crashes and restarts, the counter resets to 0, causing potential collisions.
    * *The Fix:* Store the counter on disk. To avoid heavy Disk I/O, save in **batches** (e.g., save `current_counter + 1000` to disk and only write again once those 1000 IDs are used).

---

## 🛡️ Phase 2: UUID vs. Monotonicity

### Why not just use UUID?
**UUIDs (128-bit)** are random and collision-resistant, but they have major drawbacks for high-scale systems:
* **Index Heavy:** They are large, making database indexes slow and bloated.
* **Non-Sequential:** They aren't **monotonically increasing**, meaning you cannot natively sort data by creation time.

### Why Monotonicity Matters:
IDs should be sortable so that the "Most Significant Part" (the left side) allows for easy chronological ordering. 
* **The Strategy:** `concat(time, machine_id)`. 
* **The Catch:** Even with time-first IDs, true monotonicity is hard because hardware clocks across different machines are never in perfect sync!

**If you can't keep your indexes in memory then you can't make your DB fast**

---

## 🏛️ Phase 3: Centralized ID Service

Instead of reinventing the wheel on every app server, we move generation to a **Central ID Server**.

* **Focus:** The system no longer worries about clock sync across 1000 machines, only on the ID server's state.
* **Optimization:** To reduce network latency, the ID Server hands out **batches** of IDs (e.g., "Server A, you own IDs 5000 to 6000") in a single network call.

---

## 💎 Phase 4: Existing Standards (MongoDB Object ID)

MongoDB uses a highly efficient **12-Byte (96-bit)** Hexadecimal structure that balances uniqueness, time, and performance:

| Size | Component | Purpose |
| :--- | :--- | :--- |
| **4 Bytes** | **Epoch Seconds** | Provides timestamp-based sorting (Monotonicity). |
| **5 Bytes** | **Random Value** | Unique per machine/process (Prevents collisions). |
| **3 Bytes** | **Counter** | Incremental value starting at a random number. |



---

## ⚖️ Summary of ID Strategies

| Strategy | Pros | Cons |
| :--- | :--- | :--- |
| **UUID** | No central coordination; high security. | Huge, slow indexes; not sortable. |
| **DB Auto-Increment** | Simple; strictly sequential. | Single point of failure; hard to scale. |
| **Snowflake (Twitter)** | Sortable; distributed; high performance. | Requires clock synchronization (NTP). |
| **MongoDB ID** | Compact; includes time; collision-resistant. | 96-bit is still larger than a 64-bit BigInt. |

# 🚀 ID Generation: Real-World Case Studies

How do the biggest platforms in the world generate billions of unique IDs without crashing their databases? We look at three distinct architectural patterns: **Flickr**, **Twitter**, and **Instagram**.

---

## 1. Flickr: The Centralized Ticketing Server
Flickr uses sharded MySQL databases. To avoid key collisions across different shards, they needed a way to guarantee global uniqueness.

* **The Problem:** GUIDs index poorly and are too large (128-bit). 
* **The Solution:** A centralized **Ticketing Server** that manages ID generation for the whole cluster.
* **The Logic:**
    * They use `INSERT ... ON DUPLICATE KEY UPDATE` to increment a counter. 
    * This is an **Atomic Operation** and is roughly **32x faster** than `REPLACE INTO`.
* **High Availability (The "Odd/Even" Trick):** They run **two** ticketing servers to avoid a single point of failure:
    * **Server 1:** Generates only **Even** IDs (starts at 2, increments by 2).
    * **Server 2:** Generates only **Odd** IDs (starts at 1, increments by 2).
    * A Load Balancer works in Round Robin; if one server drifts or fails, the other ensures IDs remain unique.



---

## 2. Twitter: Snowflake (Distributed 64-bit)
Twitter needed IDs for billions of tweets. Their priority was high concurrency and sortability without hitting a central database.

* **The Strategy:** Distributed ID generation where the logic lives on each **Application Server**.
* **Structure (64 bits):**
    * **41 bits:** Timestamp (ms elapsed since a custom epoch).
    * **10 bits:** Machine ID (ensures different machines don't collide).
    * **12 bits:** Per-machine sequence number (allows 4,096 IDs per ms per machine).
* **Impact:** This approach was so successful it was adopted by Discord and Instagram.



---

## 3. Instagram: Logic inside the Shard
Instagram wanted sortable 64-bit IDs but wanted to avoid "moving parts" like external ID services or Zookeeper.

* **The Innovation:** They moved the "Snowflake" logic **inside the Database** using a Stored Procedure/Function.
* **The Setup:** Each table has a default value that calls a custom function:
  `"id" bigint NOT NULL DEFAULT next_id()`

### The Logic (Stored Procedure):
The function performs bitwise operations to pack the ID:
1.  **Timestamp (41 bits):** `(now_ms - epoch) << 23`
2.  **Shard ID (13 bits):** `(shard_id << 10)` (logical shard mapping).
3.  **Sequence ID (10 bits):** The local auto-incrementing counter (`% 1024`).

```sql
-- Conceptual Instagram ID Packing
result := (now_ms - epoch) << 23;
result := result | (shard_id << 10);
result := result | seq_id;

```

---
### 📊 ID Generation Strategy Comparison

| Feature | Flickr (Ticketing) | Twitter (Snowflake) | Instagram (Stored Proc) |
| :--- | :--- | :--- | :--- |
| **Centralization** | **High** (Centralized DBs) | **None** (Fully Distributed) | **Medium** (Per DB Shard) |
| **Bit Size** | 64-bit | 64-bit | 64-bit |
| **Complexity** | **High** (Needs extra DB infra) | **Medium** (Needs Machine IDs) | **Low** (Logic stays in Table) |
| **Scalability** | Limited by LB/DB capacity | **Infinite** (Horizontal) | Tied to Physical Shards |
| **Implementation** | `REPLACE INTO` / `DUPLICATE KEY` | Application-level bit-packing | DB Stored Procedures |
