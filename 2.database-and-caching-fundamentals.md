# 🏗️ System Architecture: Building the Blogging Platform

When designing any system, we primarily focus on these **6 Key Pillars**:

1. **Database**
2. **Caching**
3. **Scaling**
4. **Delegation**
5. **Concurrency**
6. **Communication**

---

## 📝 Part 1: The Database Layer

To understand these concepts, let’s build a **Blogging Platform**. We start by defining our tables.

### The Schema
* **Users:** `id`, `name`, `bio` (Short Text)
* **Blogs:** `id`, `title`, `user_id`, `published_at`, `is_deleted`, `body` (Long Text)

### 🚨 The Power of "Soft Delete"
Instead of permanently removing data, we use an `is_deleted` flag. 
* **Recoverability:** Allows users to "undo" a deletion.
* **Archival & Audit:** Keeps a history of content for data integrity and compliance.

### ⚡ Optimization: Storing "Big AF" Content
A blog `body` is considered **Long Text**, while a `bio` is **Short Text**. 
* To improve **Seek Time**, we don't store the long text directly in the main row. 
* We store the body in a separate location and keep only a **reference** (pointer) in the original table.
* **Images:** Follow the same logic. We store the **URL** in the database, while the actual image lives in a storage bucket like **S3**.

---

## 🕒 Better Date & Time Handling
Using standard database time formats can be inefficient for heavy calculations.

* **Time:** Store as an **Epoch Integer** (seconds since 1/1/1970). This turns time into a simple mathematical value, making runtime calculations significantly faster.
* **Date:** Store in a custom `YYYYMMDD` format. It’s clean, sortable, and works like a charm for filtering.

## 🛡️ The API Layer
The user never touches the Database directly. They talk to an **API Server** which acts as a mediator.
**Reasons for this:**
* **Business Logic:** Centralized place for app rules.
* **Security:** Handles Authentication and Authorization checks.
* **Post-Processing:** Filters or formats results before they reach the user.

---

## 🚀 Making it FASTER: Caching
Caching is anything that reduces response time by saving computation. It isn't strictly RAM-based; think of it as a **glorified hash table** with advanced data structures.
* **Goal:** Maximize **Cache Hits** and minimize **Cache Misses**.

---

### 🌍 Content Delivery Network (CDN)
A CDN delivers content from the closest available server rather than hitting your main origin servers.

**Why use a CDN?**
* **Speed:** Drastically reduces response time for the user.
* **Efficiency:** Decreases the load on your main Database.
* **Uptime:** If your main server goes down, the CDN can still serve content for a while.
* **Security:** It provides a layer of protection between the public and your DB.

**What goes on a CDN?** Content that doesn't change frequently:
* Images & Videos
* API Responses
* Static Text
