# 🌐 NoSQL Databases: Scaling Beyond Tables

NoSQL databases are structured in non-tabular formats, optimized for **high scalability** and **high availability**. Systems like CosmosDB, DynamoDB, and Google Spanner are designed to scale infinitely without significant performance degradation.

---

## 🏗️ Types of NoSQL Databases

### 1. Document DB
* **Structure:** Stored in JSON/BSON format.
* **Capabilities:** Supports complex queries; secondary indexes can be created on any nested fields; partial updates to a document are possible.
* **Vibe:** The closest NoSQL equivalent to Relational DBs.
* **Examples:** MongoDB, ElasticSearch.

### 2. Key-Value (KV) Stores
* **Structure:** Simple Key and Value pairs.
* **Capabilities:** Highly partitioned (supports massive sharding); no complex queries; no partial updates.
* **Examples:** Redis, Aerospike, DynamoDB.

### 3. Columnar DB
* **Structure:** Data is stored by column rather than row.
* **Capabilities:** Reduces SCAN time; data of the same type (homogeneous) is stored together, allowing for high compression.
* **Examples:** Cassandra, HBase.

### 4. Graph DB
* **Structure:** Nodes and Edges.
* **Capabilities:** Ideal for modeling social behaviors, identifying relations/patterns, and recommendation engines (e.g., LinkedIn 1st/2nd-degree connections).
* **Examples:** Neo4j, Amazon Neptune, Dgraph, TigerGraph.



---

## 📈 Why NoSQL Scales
* **Partitioning & Sharding:** Data is distributed across multiple physical machines to prevent bottlenecks.
* **Schemaless:** Allows for rapid development where the structure can change without migrations.
* **Denormalization:** Data is optimized for the *query pattern* rather than data integrity.



---

## 🔗 The "No-JOIN" Problem: Designing Differently

In NoSQL, we don't have SQL-style `JOIN` operations. We handle this through two design patterns:

### A. Multiple Queries (The Merge Pattern)
We execute multiple queries to fetch the relevant data and then merge it at the application layer before returning the result to the client.

### B. Denormalization (The Redundancy Pattern)
We store data in redundant forms (e.g., embedding user info inside the blog document).
* **Why:** So we don't have to perform multiple DB lookups.
* **Trade-off:** Works best when the number of **Reads >>> Number of Writes**.



---

## ⚖️ SQL vs. NoSQL: When to use which?

| Feature | SQL (Relational) | NoSQL |
| :--- | :--- | :--- |
| **Integrity** | Strong ACID, high consistency | Often "Eventual Consistency" |
| **Structure** | Strictly structured | Unstructured/Schemaless |
| **Scaling** | Vertical (Scaling UP) | Horizontal (Scaling OUT) |
| **Querying** | Rich (JOINs, aggregation) | Simplified (Key lookup/range) |

> **Pro Tip:** Most production systems use **BOTH**. SQL for financial/transactional data (where ACID is mandatory) and NoSQL for user feeds, logs, or analytics (where scale is mandatory).

# ⚡ Real-time Databases & Distributed Synchronization

A Real-time Database (like **Google Firebase**) is a cloud-hosted NoSQL DB designed for instantaneous data synchronization across clients. It is document-based, serverless, and features high-speed listeners.

---

## 🚀 Key Features of Real-time DBs
* **Real-time Data Sync:** Clients "listen" for changes rather than polling the server.
* **Offline Synchronization:** Data is cached locally and synced once the connection is restored.
* **ACID Compliance:** Ensures data integrity even in a distributed environment.
* **Indexed Queries:** Optimized for high-speed retrieval.

---

## 🏗️ Building and Scaling a Real-time DB

To build our own, we need to handle a **KV Store**, **Subscriptions**, and **Conflict Resolution** (both online and offline).

### 1. The Connection: WebSockets vs. SSE
We use **WebSockets** instead of Server-Sent Events (SSE). 
* **WebSockets:** Full-duplex (two-way) communication. Essential for real-time broadcasting where clients both send and receive data.
* **SSE:** One-way communication (server to client only).

### 2. The Scaling Challenge
A single API server can only handle a few thousand WebSocket connections. To scale to millions, we must distribute clients across multiple API servers. However, if **User A** is on **Server 1** and **User B** is on **Server 2**, they cannot "talk" directly.



### 3. The Solution: Redis Pub/Sub as the "Hub"
We use **Redis** as a Message Broker to connect all API servers together.

* **The Workflow:**
    1.  Client A sends a message to **API Server 1**.
    2.  API Server 1 **PUBLISHES** that message to a specific Redis Channel.
    3.  Every other API Server is **SUBSCRIBED** to that Redis Channel.
    4.  They receive the message and push it down to their own connected clients.



---

## 🛠️ Redis Pub/Sub Commands

Redis acts as the central post office for our real-time updates:

| Command | Description |
| :--- | :--- |
| `SUBSCRIBE [channel]` | The API server starts listening for any updates on a specific topic/group. |
| `PUBLISH [channel] [message]` | The API server pushes a new update into the hub for everyone else to see. |
| `UNSUBSCRIBE [channel]` | The API server stops listening (usually when a client disconnects). |

---

## 🛡️ Handling Conflicts: Online vs. Offline
* **Online Conflicts:** Handled via the **Pessimistic/Optimistic Locking** strategies we discussed in the KV Store implementation.
* **Offline Synchronization:** When a client reconnects, it sends a "Delta" (the changes made while offline). The server compares timestamps or version numbers to merge the data or resolve conflicts.
