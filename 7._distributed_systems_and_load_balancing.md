# 🌐 Distributed Systems Foundations

A **Distributed System** consists of multiple components located on different machines that communicate and coordinate their actions. To the end-user, it appears as a **single coherent system**.

### ⚖️ The Trade-offs
* **Why use them?** Horizontal scaling, reliability (no single point of failure), and performance.
* **Why avoid them?** Increased latency (network overhead) and complex observability.

---

## ⚖️ Load Balancers: The Engine of Scalability

The **Load Balancer (LB)** is the primary reason systems can scale. It sits between the user and the backend, distributing traffic to prevent any single server from becoming a bottleneck.

### 🛠️ Load Balancing Algorithms
* **Round Robin / Weighted RR:** Sequential distribution based on capacity.
* **Least Connections:** Picks the server with the fewest active tasks.
* **IP / URL Hash:** Ensures request stickiness based on client identity or destination.

---

## 🏗️ Designing a High-Availability Load Balancer

### Key Components
1.  **LB Servers:** The actual nodes (e.g., `10.0.0.1`, `10.0.0.17`) that receive and forward traffic.
2.  **Backend (B.) Servers:** The worker nodes running the actual service (e.g., **Payments Service**).
3.  **LB Registry:** A database or configuration file that stores the list of healthy backend IPs.
4.  **Orchestrator:** The brain of the infrastructure.
    * **Health Checks / Heartbeats:** It constantly pings backend servers to ensure they are alive.
    * **Sync:** It pushes updates to all LB servers whenever a backend server is added or removed.
    * **Dev Console:** An interface for developers to manually manage the infrastructure.


---

## 📈 Scaling the Load Balancer (The DNS Layer)

As seen in the payments architecture, we don't just have one Load Balancer. We scale the LBs themselves horizontally.

### The DNS Resolution Flow:
1.  **User Request:** The user tries to access `lb.payments.google.com`.
2.  **DNS (CoreDNS / Route53):** The DNS holds a list of multiple LB Server IPs (e.g., `10.0.0.1`, `10.0.0.17`, `10.0.0.21`).
3.  **Resolution:** DNS **resolves to one of them** (usually via Round Robin), sending the user to a specific LB Server.
4.  **Forwarding:** The chosen LB Server forwards the request to one of the many Backend Servers in the **Payments Service** cluster.
5.  **Persistence:** The Backend Servers finally interact with the **Payments DB**.


---

> **Design Note:** By keeping a local "LB Registry" synced on each LB server, we ensure that even if the Orchestrator or the main Registry DB goes down, the existing LB servers can still route traffic to the backends.

# 🔒 Remote and Distributed Locking

In a distributed environment, traditional synchronization (like Mutexes or Semaphores) fails because memory is not shared across different physical machines. We need **Remote Locks**—managed by a central Lock Manager—to coordinate access to shared resources.

---

## 🏗️ Evolution of Synchronization

The tool we use depends entirely on the scope of the "Critical Section" we are protecting:

| Scenario | Synchronization Tool | Scope |
| :--- | :--- | :--- |
| **Multiple Threads** | Mutex / Semaphores | Single Process / Single RAM |
| **Multiple Processes** | Common Storage (Shared Disk/File) | Single Machine |
| **Multiple Machines** | **Remote Locks** | Entire Distributed Network |

---

## 🛠️ Requirements for a Robust Remote Lock

To prevent data corruption in systems like a **Distributed Queue**, a remote lock must follow these three principles:

1.  **Shared Visibility:** Every machine in the cluster must be able to see and communicate with the Lock Manager.
2.  **Atomicity:** The "Check and Set" operation must be atomic. There can be no gap between checking if a lock is free and claiming it (no context switching interference).
3.  **Fault Tolerance (TTL):** Every lock must have a **Time-To-Live**. If a machine acquires a lock and then crashes, the lock will automatically expire, preventing a permanent "deadlock" state.



---

## 🔄 Implementation Logic: The "Try-Lock" Pattern

When a machine needs to access a shared resource (like the front of a remote queue), it follows this workflow:

### 1. The Atomic Acquisition
The machine attempts to write a unique key to the manager (e.g., Redis). 
* **The Rule:** "Write only if the key does NOT already exist." 
* In Redis, this is the `SET NX` command.

### 2. Guarding the Critical Section
If the write succeeds, the machine holds the lock. While held, any other machine attempting to "Try-Lock" will be denied by the manager.

### 3. Automatic Release (TTL)
If the machine fails to release the lock (due to a crash or network timeout), the manager deletes the key after the TTL expires. This ensures the system stays "alive" and other machines can eventually proceed.



---

## 🚦 Use Case: Remote Queue (No Front Protection)

Imagine a shared queue where multiple machines want to `REMOVE` the front item.

* **Without Locks:** Machine A and Machine B might both read the "Front" at the exact same time and both try to process the same data.
* **With Remote Locks:** 1. Machine A acquires `lock_queue_front`.
    2. Machine A removes the item.
    3. Machine B tries to acquire `lock_queue_front` but is rejected.
    4. Machine A finishes and deletes the lock.
    5. Machine B can now proceed to the *next* item.
