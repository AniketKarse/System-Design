# 🌐 Distributed Systems Foundations

A **Distributed System** consists of multiple components located on different machines that communicate and coordinate their actions. To the end-user, it appears as a **single coherent system**.

### ⚖️ The Trade-offs
* **Why use them?** Horizontal scaling, reliability (no single point of failure), and performance.
* **Why avoid them?** Increased latency (network overhead) and complex observability.

---

## ⚖️ Load Balancers: The Engine of Scalability

The **Load Balancer (LB)** is the primary reason systems can scale. It sits between the user and the backend, distributing traffic to prevent any single server from becoming a bottleneck.

### 🛠️ Load Balancing Algorithms
* **Round Robin / Weighted RR:** Sequential distribution based on capacity.
* **Least Connections:** Picks the server with the fewest active tasks.
* **IP / URL Hash:** Ensures request stickiness based on client identity or destination.

---

## 🏗️ Designing a High-Availability Load Balancer

### Key Components
1.  **LB Servers:** The actual nodes (e.g., `10.0.0.1`, `10.0.0.17`) that receive and forward traffic.
2.  **Backend (B.) Servers:** The worker nodes running the actual service (e.g., **Payments Service**).
3.  **LB Registry:** A database or configuration file that stores the list of healthy backend IPs.
4.  **Orchestrator:** The brain of the infrastructure.
    * **Health Checks / Heartbeats:** It constantly pings backend servers to ensure they are alive.
    * **Sync:** It pushes updates to all LB servers whenever a backend server is added or removed.
    * **Dev Console:** An interface for developers to manually manage the infrastructure.

## 📈 Scaling the Load Balancer (The DNS Layer)

As seen in the payments architecture, we don't just have one Load Balancer. We scale the LBs themselves horizontally.

### The DNS Resolution Flow:
1.  **User Request:** The user tries to access `lb.payments.google.com`.
2.  **DNS (CoreDNS / Route53):** The DNS holds a list of multiple LB Server IPs (e.g., `10.0.0.1`, `10.0.0.17`, `10.0.0.21`).
3.  **Resolution:** DNS **resolves to one of them** (usually via Round Robin), sending the user to a specific LB Server.
4.  **Forwarding:** The chosen LB Server forwards the request to one of the many Backend Servers in the **Payments Service** cluster.
5.  **Persistence:** The Backend Servers finally interact with the **Payments DB**.


---

> **Design Note:** By keeping a local "LB Registry" synced on each LB server, we ensure that even if the Orchestrator or the main Registry DB goes down, the existing LB servers can still route traffic to the backends.
