# 🏎️ Concurrency: Managing Simultaneous Actions

In a distributed system, concurrency happens when multiple processes try to access or modify the same data at the exact same time.

## 🤝 The "Clap" Problem
Imagine a blog has 10 claps. Two users click "Clap" at the same microsecond.
1. **User A** reads `10`.
2. **User B** reads `10`.
3. **User A** updates it to `11`.
4. **User B** updates it to `11`.
**Result:** One clap is lost! This is a classic "Race Condition."

---

## 🛠️ How to handle it

### 1. Locks
We can use locking mechanisms to ensure only one "thing" touches the data at a time:
* **Mutex (Mutual Exclusion):** Only one thread can acquire the lock. Others must wait until it's released.
* **Semaphores:** Allows a specific number of threads to access a resource.

### 2. Lock-free Data Structures
Specialized data structures designed to handle concurrent access without the overhead of traditional locks, often using hardware-level instructions.

### 3. Atomic Operations
To handle claps correctly, the increment must be **Atomic** (it happens as a single, indivisible step).
* **DB Transactions:** If the Database permits, perform the increment within a transaction (e.g., `UPDATE blogs SET claps = claps + 1`).
* **Atomic Counters:** Use tools like Redis `INCR` that are built to be atomic by nature.

# 🗣️ Communication: Connection Patterns

Systems are a web of connections. Understanding how these components talk is vital for performance and reliability.

## 🔄 Types of Communication
* **Client ↔ Server:** The standard request-response model.
* **Client ↔ Client:** Peer-to-Peer (P2P).
* **Inter-Service:** Communication between different microservices.
* **Intra-Service:** Communication within different parts of the same service.
* **Inter-Process:** Communication between two different running programs.
* **Intra-Process:** Communication within a single program.

The usual standard is **REST** (Client sends a request, Server sends a response).

---

## 📡 Real-time Data Transfer (The Cricbuzz Example)

How does a live score reach your screen? We have three main strategies:

### 1. HTTP Short Polling (Ajax Polling)
The client pings the server at fixed intervals (e.g., every 5 seconds) asking for updates.
* **The Flaw:** Wasteful. During a "drinks break" when no runs are scored, the client still hits the server, wasting resources.



### 2. HTTP Long Polling
The client sends a request, but the server **holds the connection open**. It only responds when there is *new* data.
* **The Benefit:** No empty responses.
* **The Flaw:** Can cause **timeouts** if the data doesn't change for a long period.



### 3. WebSockets
A persistent, two-way connection is established and kept open.
* **The "Lit" Feature:** Instead of the client asking, the **server proactively pushes** data to the client.
* **The Vibe:** True real-time synchronization.
