# 🗄️ Relational Database Deep Dive

When building industrial-grade systems, the focus is majorly on **Relational Databases (RDBMS)**. These provide the structure and reliability needed for complex applications.

---

## 📐 Standards & Constraints

In the industry, we typically normalize data up to **3NF (Third Normal Form)** to ensure efficiency and reduce redundancy. To maintain data integrity, we use:

* **UNIQUE:** Ensures no two rows have the same value in a column.
* **NOT NULL:** Ensures a column cannot be empty.
* **PRIMARY KEY:** A unique identifier for every record.
* **FOREIGN KEY:** Maintains a link between data in two different tables.

---

## 🛡️ ACID Properties
Every transaction in a relational database must satisfy these four properties to ensure data reliability:
1.  **Atomicity:** The entire transaction succeeds or it all fails.
2.  **Consistency:** Data must meet all validation rules after a transaction.
3.  **Isolation:** Transactions do not interfere with each other.
4.  **Durability:** Once committed, data is saved even in a system crash.

---

## ⚡ Database Indexes
The primary job of an index is to **improve lookup speed**.

### How it works:
We create a new **Index Table** that is sorted by the specific column we intend to search (e.g., `user_id` or `email`). 
* This index table takes significantly less memory than the original table.
* Because the index is sorted, the database can perform a **Binary Search** instead of scanning the entire original table.



### The Trade-off: Write Performance
While indexes make searching fast, they **decrease write performance**.
* Every time you insert or update data, the database must also update the index table.
* Since the index must remain sorted, reorganizing it is a costly operation.

## 🔒 Database Locking
Locks are used to manage concurrency and prevent data corruption when multiple users access the same row.

| Lock Type | Also Known As | Rules |
| :--- | :--- | :--- |
| **Read Lock** | Shared Lock | Reserved for reading. Others can read, but **cannot write**. |
| **Write Lock** | Exclusive Lock | Reserved for writing. Others **cannot read or write**. |

### Locking Syntax
* **Locking for Read:** `SELECT ... FOR SHARE;`
* **Locking for Write:** `SELECT ... FOR UPDATE;`

### Speeding up Traversal: SKIP LOCKED & NOWAIT
* **SKIP LOCKED:** Skips any rows that are currently locked by another transaction. This prevents the system from "hanging" or waiting.
* **NOWAIT:** Instead of waiting for a lock to be released, the transaction aborts immediately and returns an error (ERROR 3572).

---

## ✈️ Case Study: Airline Check-in System

To handle thousands of passengers checking in simultaneously, we designed a high-concurrency system:

1.  We use `FOR UPDATE` to lock specific seats while they are being assigned to a passenger.
2.  We use `SKIP LOCKED` to speed up the process. 
    * If Transaction A is currently locking seat 1A, Transaction B doesn't wait for it.
    * Transaction B simply **skips** the locked row and moves to the next available seat.
    * This ensures that the check-in process never has to wait for other transactions to finish.
