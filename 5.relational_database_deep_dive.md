# 🗄️ Relational Database Deep Dive

When building industrial-grade systems, the focus is majorly on **Relational Databases (RDBMS)**. These provide the structure and reliability needed for complex applications.

---

## 📐 Part 1: RDBMS Standards & Constraints

In the industry, we typically normalize data up to **3NF (Third Normal Form)** to ensure efficiency and reduce redundancy. To maintain data integrity, we use:

* **UNIQUE:** Ensures no two rows have the same value in a column.
* **NOT NULL:** Ensures a column cannot be empty.
* **PRIMARY KEY:** A unique identifier for every record.
* **FOREIGN KEY:** Maintains a link between data in two different tables.

---

## 🛡️ ACID Properties
Every transaction in a relational database must satisfy these four properties to ensure data reliability:
1.  **Atomicity:** The entire transaction succeeds or it all fails.
2.  **Consistency:** Data must meet all validation rules after a transaction.
3.  **Isolation:** Transactions do not interfere with each other.
4.  **Durability:** Once committed, data is saved even in a system crash.

---

## ⚡ Database Indexes
The primary job of an index is to **improve lookup speed**.

### How it works:
We create a new **Index Table** that is sorted by the specific column we intend to search (e.g., `user_id` or `email`). 
* This index table takes significantly less memory than the original table.
* Because the index is sorted, the database can perform a **Binary Search** instead of scanning the entire original table.



### The Trade-off: Write Performance
While indexes make searching fast, they **decrease write performance**.
* Every time you insert or update data, the database must also update the index table.
* Since the index must remain sorted, reorganizing it is a costly operation.

---

## 🔒 Database Locking
Locks are used to manage concurrency and prevent data corruption when multiple users access the same row.

| Lock Type | Also Known As | Rules |
| :--- | :--- | :--- |
| **Read Lock** | Shared Lock | Reserved for reading. Others can read, but **cannot write**. |
| **Write Lock** | Exclusive Lock | Reserved for writing. Others **cannot read or write**. |

### Locking Syntax
* **Locking for Read:** `SELECT ... FOR SHARE;`
* **Locking for Write:** `SELECT ... FOR UPDATE;`

### Speeding up Traversal: SKIP LOCKED & NOWAIT
* **SKIP LOCKED:** Skips any rows that are currently locked by another transaction. This prevents the system from "hanging" or waiting.
* **NOWAIT:** Instead of waiting for a lock to be released, the transaction aborts immediately and returns an error (ERROR 3572).

---

## ✈️ Case Study: Airline Check-in System

To handle thousands of passengers checking in simultaneously, we designed a high-concurrency system:

1.  We use `FOR UPDATE` to lock specific seats while they are being assigned to a passenger.
2.  We use `SKIP LOCKED` to speed up the process. 
    * If Transaction A is currently locking seat 1A, Transaction B doesn't wait for it.
    * Transaction B simply **skips** the locked row and moves to the next available seat.
    * This ensures that the check-in process never has to wait for other transactions to finish.

## 🔑 Part 2: Designing a SQL-Backed KV Store

A Key-Value store is the simplest base for any DB use case. Understanding it helps in understanding transactions, locking, and read patterns.

### 🏗️ The Schema
* **Key:** (Primary Key) 
* **Value:** (Blob) The data.
* **TTL:** (Epoch Int) Time-to-live. (We use **-2** for Soft Delete).

### 🛠️ Core Operations
1. **GET:** `SELECT * FROM store WHERE key = @key AND ttl > now();`
2. **DEL:** `UPDATE store SET ttl = -2 WHERE key = @key AND ttl > now();`
3. **PUT:** An **Upsert** (Update if exists, Insert if not).
4. ####  The Cleanup (Garbage Collection)
Because we use **Soft Deletes** (TTL = -2) and expired timestamps, the table will naturally "bloat" over time, slowing down indexes. We solve this with a **One-Shot Cleanup** process.

* **The Logic:** `DELETE FROM store WHERE ttl < now();` 
    * *Note: Since -2 is less than the current epoch/timestamp, this single query clears both expired data and soft-deleted records.*
* **Frequency:** This background task runs at a tunable interval—typically every **5 minutes**.
* **Purpose:** It physically purges "dead" rows from the disk. This reclaims storage space and prevents the index tree from becoming fragmented, ensuring that **GET** operations remain high-performance.

### 🏎️ Handling Race Conditions in PUT
When multiple users update the same key, we choose one of three strategies:

1. **Last Write Wins (LWW):** No locking; the final request overwrites everything.
2. **Pessimistic Locking:** `FOR UPDATE NOWAIT;` allows only one update at a time.
3. **Optimistic Locking:** "Update what you read." No locking; you only write if the TTL/Version hasn't changed since you last read it.

# ⏳ Implementing TTL (Time-To-Live)

In a Key-Value store, managing memory and storage is just as important as the data itself. If we don't delete expired keys, our database will grow indefinitely (bloat), slowing down every other operation.

---

## 🛠️ Three Strategies for TTL

To build a robust system, we combine different approaches to balance **accuracy** and **performance**.

### 1. The Active Cleanup (CRON Job)
A background process runs every **t minutes** to perform a "One-Shot" purge of the database.
* **The Logic:** `DELETE FROM store WHERE ttl < now();`
* **The Issue:** Until the next interval (t) triggers, expired keys are still physically in the DB and taking up space.
* **Frequency:** Usually tunable (e.g., every 5 minutes).

### 2. The Lazy Delete (On-Access)
We don't wait for a background job. Instead, we check expiration during the **GET** operation.
* **The Logic:** When a user requests a key, the system checks the TTL. If `ttl < now()`, the system deletes the key immediately and returns a "Not Found" error.
* **The Issue:** If a key is **never accessed** again, it will never be deleted. In a high-volume system, this leads to millions of "zombie" rows that Lazy Delete can't reach.



### 3. The Probabilistic Cleanup (The "Redis" Way)
To keep the database lean without scanning every single row (which is costly), we use a sampling strategy based on the **Central Limit Theorem**:

1.  **Sample:** Pick **20 random keys** that have a TTL set.
2.  **Purge:** Check their timestamps and delete all keys that have already expired.
3.  **The Threshold:** If more than **25%** (5 keys) were expired, it suggests the database is currently full of "junk." 
4.  **Repeat:** If the 25% threshold is hit, **repeat the process immediately**. If not, stop and wait for the next scheduled run.



---

## 🎯 Summary Table

| Strategy | Speed | Storage Efficiency | Risk |
| :--- | :--- | :--- | :--- |
| **CRON Job** | High | Medium | Data exists until the next run. |
| **Lazy Delete** | High | Low | Never-accessed keys cause bloat. |
| **Probabilistic** | Balanced | High | Theoretically, some expired keys stay longer. |
